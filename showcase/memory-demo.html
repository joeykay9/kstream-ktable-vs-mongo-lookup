How to read this: KTable-Co and KTable back their state store with RocksDB (native off-heap memory). RocksDB adds significant process overhead even at small dataset sizes — JNI, block cache, memtables, and SST file metadata all contribute to RSS regardless of how much data is actually stored. GlobalKTable, by contrast, defaults to an in-memory key-value store (Java heap, no RocksDB). Its RSS is lower at small-to-medium dataset sizes precisely because there is no RocksDB overhead. The tradeoff flips at large dataset sizes: the full dataset must fit in the JVM heap for GlobalKTable (every instance holds all records), while KTable strategies hold only dataset ÷ partition count per instance and RocksDB handles data beyond the memtable on disk. Mongo-Batch and Mongo-Sync hold no reference data in-process at all — their RSS is a flat baseline (JVM + MongoDB driver).

RSS is measured via ps -o rss= after a forced GC and a 2-second settle period. RocksDB live-data-size reflects SST files on disk (0 means all data fits in the in-memory memtable, which is typical below ~600k products with default settings).
